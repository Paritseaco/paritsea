# Paritsea Semantic Constitution

## 0. Purpose

This document defines the highest-level semantic rules of Paritsea.

It overrides all other documentation in case of contradiction, ambiguity, or interpretive conflict.

All system behavior, UX, language design, and roadmap execution must comply with this constitution.

---

## 1. Ontological Structure of Paritsea

Paritsea operates on two independent but connected systems:

### 1.1 Model A — Structural Pipeline (Truth Layer)

Defines how the system is architecturally organized.

- Journal → System → Implementation
- Governs pages, routing, navigation, and content placement
- Represents the structural reality of the product

This is the **system architecture truth**.

---

### 1.2 Model B — Language Hierarchy (Cognitive Layer)

Defines how language is exposed to users over time.

- Entry Vocabulary (Surface)
- Gateway Vocabulary (Transition)
- Foundational Ontology (Core)

This is the **cognitive perception system**, not the structural system.

---

## 2. Authority Hierarchy

In case of conflict:

1. Semantic Constitution (this document)
2. Model A (Structural Pipeline)
3. Model B (Language Hierarchy)
4. UX / Copy / Implementation suggestions
5. Roadmap (execution plan)

---

## 3. Vocabulary Classification Rule

All system terms must be classified into one of the following:

- Entry Layer → immediately understandable UI language
- Gateway Layer → transitional reframing language
- Core Layer → structural ontology concepts

Pipeline terms (Journal / System / Implementation):
→ classified as Entry Vocabulary in user-facing UI

---

## 4. Language Integrity Rule

- Core vocabulary MUST NOT appear in onboarding or first-touch UI.
- Gateway vocabulary MUST NOT be used as explanatory replacement for Core concepts.
- Entry vocabulary MUST NOT be overloaded with abstract meaning.

Language must preserve cognitive progression:
Entry → Gateway → Core

---

## 5. Roadmap Constraint Rule

Roadmap is an execution plan, not a truth system.

- Roadmap must always conform to Model A and Model B
- Roadmap cannot redefine ontology
- Roadmap must be updated when system state changes, not the other way around

---

## 6. Conflict Resolution Rule

When contradictions appear:

- First resolve against Semantic Constitution
- Then align with Model A
- Then adjust Model B
- Only then update UX or Roadmap

Never resolve conflicts in UX layer alone.

---

## 7. System Philosophy

Paritsea does not optimize communication.

Paritsea structures cognition through controlled linguistic exposure over time.

Language is not decoration.

Language is infrastructure of thought.

---

## 8. Historical Integrity Rule

The system must preserve historical execution traceability.

- Completed phases MUST NOT be deleted from documentation
- Completed phases MAY be collapsed into summary form, but original structure must remain accessible
- All removals must be explicitly marked as "archived", not erased

Rationale:
Paritsea requires cognitive auditability. Loss of historical structure results in loss of system explainability.

---

## 9. Edit Authority Constraint

AI is not allowed to independently determine:

- whether a system artifact should be deleted
- whether a phase is "irrelevant"
- whether historical content should be removed entirely

AI may:
- suggest pruning
- propose restructuring
- flag redundancy

But final structural removal requires explicit user confirmation or declared rule in Semantic Constitution.

---

## 10. Execution Contract Rule

The AI must treat `docs/roadmap.md` as an immutable execution specification.

### 10.1 Prohibitions

AI must NOT:
- Reinterpret phase meaning
- Collapse, merge, or remove phases
- Introduce new phases or restructure roadmap
- Modify the roadmap structure without explicit user instruction

### 10.2 Permitted Actions

AI may only:
- Execute tasks exactly as written in the roadmap
- Report missing implementation
- Flag contradictions without resolving them
- Suggest clarifications (not changes)

### 10.3 Structural Authority

All structural changes to the roadmap require:
- Explicit user instruction, OR
- A new rule declared in this Semantic Constitution

**Roadmap is not a narrative.**

**Roadmap is a contract.**
