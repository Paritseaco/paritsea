# Paritsea — Content Governance System
**Status:** Production. Source of truth for all content decisions.
**Last reviewed:** 2026-06

---

## Foundational principle

Every page sits at one **altitude** on the seeing → structuring → doing spine. Content drifts when it is written at the wrong altitude for its page. The whole system stays coherent if one rule holds:

> **Write each page at its altitude. Never borrow a neighbor's altitude to sound richer.**

### Three altitudes

| Altitude | Pages | Character |
|----------|-------|-----------|
| **Seeing** | Journal, Journal Post | Interpretation, ungoverned, human |
| **Structuring** | System, Framework, Protocol, Standard | Formalized, governed, precise |
| **Doing** | Implementation | Applied, evidential |
| **Support** | About, Licensing, Contact | Context, governance, conversation |

## Cross-page content boundaries

| Boundary | Rule of separation |
|----------|--------------------|
| **Journal vs System** | Journal = *ungoverned observation*. System = *formalized rule*. The moment an observation prescribes, it has left Journal. |
| **Framework vs Protocol/Standard** | Framework = *immutable foundation*. Protocol/Standard = *derived & versioned*. Anything that can change is not Framework. |
| **Protocol vs Standard** | Protocol = *obligation* (what must hold). Standard = *threshold* (how it's measured). Never let one contain the other. |
| **System vs Implementation** | System = *the rule*. Implementation = *the rule running in reality*. Examples-as-proof belong to Implementation; rules belong to System. |
| **Home vs everything** | Home only *frames and routes*. It never hosts what a deeper page owns. |
| **About vs Journal** | About = *origin of the lens*. Journal = *the lens in use*. Founder reflection that observes the world → Journal, not About. |
| **Licensing vs System** | System states the knowledge; Licensing states permission over it. Governance of *ideas* ≠ governance of *use*. |

## Anti-drift rules

1. **One altitude per page.** If a paragraph would fit better one layer up or down, move it there.
2. **No upward borrowing.** A Standard must not philosophize to feel important; a Protocol must not narrate.
3. **No downward borrowing.** Framework/Journal must not smuggle in specs or how-to to seem useful.
4. **Bridges, not merges.** Cross-layer links are allowed; cross-layer *content* is not. Link to the neighbor; don't reproduce it.
5. **Immutability is sacred.** Framework never gains a changelog. Versioning lives only in Protocol/Standard.
6. **CTA discipline.** Commercial/contact CTAs appear only on Implementation, Licensing, Contact. Never on Journal/Framework/Standard.
7. **No marketing register anywhere.** No urgency, scarcity, testimonials, or feature-selling on any page.
8. **Plain-language precedes formal** on governance pages — always.

## Editorial governance rules

- **Single authorial voice.** Paritsea is one person's IP and one lens — no committee tone, no ghost-multiplicity.
- **Altitude check before publish.** Every new piece answers: *what one question does this answer, and is this the page that owns that question?* If two pages claim it, one is wrong.
- **Derivation integrity.** New Protocols/Standards must not contradict Framework. Framework changes are effectively prohibited; if it must change, it is a new foundational act, not an edit.
- **One-question rule.** A page answers exactly one core user question. If it starts answering two, split the content to the page that owns the second question.
- **Bridge budget.** A reading page (Journal Post, Framework, Standard) carries at most **one** outbound vertical bridge. Lateral links may be plural.
- **Language parity.** TH and EN must carry identical altitude and meaning; translation never upgrades or downgrades register.
- **Review cadence.** Audit annually for drift using the classification rules below; flag any page answering a question it "must not answer."

## Future content classification rules

When new content appears, route it by asking, in order:

**1.** Is it an observation of reality? → **Journal**

**2.** Does it prescribe or formalize?
- Foundational & immutable → **Framework**
- An obligation → **Protocol**
- A measurable threshold → **Standard**

**3.** Is it the system applied in a real product? → **Implementation**

**4.** Is it about origin / the person behind the lens? → **About**

**5.** Is it about permission / use / IP? → **Licensing**

**6.** Is it a request for conversation? → **Contact**

### Tie-breakers

| Conflict | Resolution |
|----------|-----------|
| Journal *or* System | "Is it governed?" — Governed → System; not yet → Journal |
| System *or* Implementation | "Is it the rule or the running of it?" |
| Protocol *or* Standard | "Obligation or threshold?" |
| Fits nowhere cleanly | **Do not create a new page type.** Reframe to an existing altitude or hold back. The page taxonomy is closed. |

## The governing sentence

> *Does this content answer the one question this page owns — at this page's altitude — without borrowing from its neighbors?*

If yes, publish. If no, move it or hold it.

This sentence applies to every piece of content, every edit, and every future addition across all pages.

## Language Governance System

Content in Paritsea is governed by a 3-layer language model:

### Layer Access Control

- Entry Vocabulary: unrestricted
- Gateway Vocabulary: contextual use only
- Foundational Ontology: restricted to deep/system content

### Authorization Rule

Core vocabulary may only be used when:
- Explaining system structure
- Designing cognitive frameworks
- Writing deep theoretical content

### Enforcement Principle

Misuse of Core vocabulary in UI contexts is considered:
- Cognitive overload risk
- Structural inconsistency

### Governance Intent

This is not a style guide.

It is a control system for cognitive progression.
